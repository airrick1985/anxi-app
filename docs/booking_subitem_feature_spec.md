# 預約系統：多層次日期時段與名額控管 (Sub-Menu Quota) 設計規格

## 1. 需求背景與目標
- **原有限制**：先前的名額與日期時段，都是綁定在第一層的「預約項目」(例如：對保、初驗)。
- **新需求**：需要根據子選單的更深層次（例如第三層選項）來決定「日期、時段」以及「名額」。
  - **情境舉例**：用戶選擇「對保」 > 「需貸款」 > (展開第三層單選) 「台灣銀行」、「土地銀行」。
  - 後台必須能針對「台灣銀行」設定它專屬的可預約日期、時段與名額上限；「土地銀行」也有自己獨立的日期、時段與名額。
- **原則**：必須向下相容，尚未啟用子選單的舊有設定，維持原本的「預約項目」全域時段與名額計算方式。

---

## 2. 後台管理端 (BookingRuleManager.vue) 設計

### A. 選單結構擴充 (Menu Level 3)
1. **設定位置**：在「預約系統狀態設定」 > 「預約選單與人員」的頁籤中。
2. **功能新增**：
   - 針對任意「選擇方式」(例如「需貸款」)，新增一個【設定子選單 (Sub-Options)】功能。
   - 管理員可加入多個子選單選項名稱 (例如：`台灣銀行`, `土地銀行`, `第一銀行`)。
   - 資料結構：在 `projectSettings.bookingMenu` 內的 `methods` 物件中，新增 `subOptions: ['台灣銀行', '土地銀行']` 的字串陣列。

### B. 批次與時段設定 (Batch & Slots Configuration)
1. **設定位置**：在「批次管理」新增或編輯批次的「時段設定」介面。
2. **功能新增 (分配子項目名額)**：
   - 當該批次所屬的「預約項目」旗下有設定子選單時，在勾選/建立每一天的「時段」(e.g. `10:00-11:00`) 時，除了畫面上原有的「時段總名額」外，提供一個**【展開子選項名額設定】**的開關。
   - 展開後，會列出該項目下所有的子選項 (台灣銀行、土地銀行...)，允許管理員輸入各自獨立的名額 (例如：台灣銀行 2人、土地銀行 1人)。
   - **彈性日期設定**：如果某天 (例如 5/1) 只有台灣銀行有人員駐點，則在該日子的時段，將台灣銀行的名額設為 `> 0`，其他銀行的名額設為 `0` 即可。

3. **資料結構擴充 (相容性)**：
   - 既有的 batch slot 結構為 `{ time: '10:00', limit: 10 }`。
   - 擴充後的結構為：
     ```json
     {
       "time": "10:00-11:00",
       "limit": 10,
       "subOptionLimits": {
         "台灣銀行": 2,
         "土地銀行": 1
       }
     }
     ```
   - *向下相容：若沒有 `subOptionLimits`，則代表該時段不區分銀行，任何符合該項目的選項皆可預約，共用 `limit` 容量。*

## 3. 前端預約端 (BookingPage.vue) 設計

### A. 選擇流程調整 (Step 1 -> Step 2)
- **Step 1 (預約資料填寫)**：
  - 用戶選擇「預約項目」(對保) -> 「選擇方式」(需貸款)。
  - 若該「選擇方式」具有 `subOptions`(子選項)，動態渲染出第三層單選的下拉選單或選項按鈕 (例如請選擇：台灣銀行/土地銀行)。
  - **必填驗證**：用戶必須選定子選項後才能進入下一步。
  - 將選定的選項存入 `formStep1.subOption`。

- **Step 2 (日期與時段選擇過濾)**：
  - 進入日期選擇器前，前端會持著用戶所選的 `subOption` (例如：台灣銀行) 來過濾可選日期。
  - **日期過濾 (Date Picker)**：只有包含「台灣銀行名額 > 0」 或 「未設定子項限制的共用時段」 的日子，兩者滿足其一才會在日曆上開放點擊 (allowed-dates)。
  - **時段過濾 (Time Slots)**：選定日期後，只顯示該日中「台灣銀行名額未滿」或「共用名額未滿」的時段。顯示時，時段的剩餘名額提示將依據子項目的剩餘名額 (若有設定) 來呈現。

### B. 最終預約與扣減 (Submit)
- **資料送出**：將選定的 `subOption` 和原本的表單一併送出至 Firestore。
- **預約紀錄顯示**：在 Step 3 預約成功畫面、以及後段查詢預約紀錄時，將第三層子選項明確標示 (例如：對保 - 需貸款 - [台灣銀行])。

---

## 4. 後端 (Cloud Functions / Firebase) 調整
當使用者送出包含 `subOption` 的預約請求時，檢查名額 (Quota Check) 邏輯需進行改寫：
1. **計算名額規則**：
   - 檢查目標時段 (Target Time Slot) 的 `limit` (總名額) 是否 > bookedCount。
   - 若該時段有設定 `subOptionLimits`，且使用者有提交 `subOption`：
     - 去查出該時段目前「所屬此 subOption」的預約人數 (`bookedCountBySubOption`)。
     - 若 `bookedCountBySubOption[台灣銀行]` >= `subOptionLimits[台灣銀行]`，則回傳該子項目名額已滿的錯誤拒絕預約。
2. **寫入更新**：利用 Transaction，除更新該時段的總預約數，也一併更新該時段下給定 `subOption` 的預約計數，防止超賣。

## 技術與邏輯確認 (待您回覆釐清)
1. **表單頁面階段性**：目前用戶是在 `Step 1` 選擇方式 (Method)，`Step 2` 選擇日期與時段。您能接受將這第三層的子選單(銀行選項) 安排在 `Step 1` 中，以便用戶選定後，系統能在 `Step 2` 自動過濾出該銀行有開放的專屬日期嗎？
2. **名額加總行為**：如果設定 10:00 時段，台灣銀行提供 2 個名額、土地銀行提供 2 個名額。該時段的總名額上限 (limit) 一般會自動設定為 `2 + 2 = 4`，還是說總名額可能只有 3，且先搶先贏？(建議：將各子項目名額相加等於總時段名額會比較直觀)。
